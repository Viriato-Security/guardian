# What Is Guardian

Guardian is a kernel-level eBPF observability agent for AI systems. It attaches to the Linux kernel, intercepts syscalls made by AI model processes, enriches each raw event with identity metadata, signs the resulting event stream with a cryptographic hash chain, and forwards it to the Viriato platform for storage, interpretation, and compliance reporting.

One process. One YAML file. Zero changes to AI model code.

---

## The Core Idea in One Sentence

Guardian captures facts about what AI models do at the operating system level and produces a tamper-evident record of those facts — it does not decide what those facts mean.

---

## "Dumb by Design"

Guardian is deliberately simple in scope. It observes and records. It does not analyse, score, classify, or recommend.

This is not a limitation — it is the design principle that makes Guardian useful for compliance. When an auditor asks "what did this AI model actually do on March 14th at 14:32 UTC?", the answer must come from a system that was capturing raw facts at the time, not from a system that was making inferences. An inference engine can be configured to produce any answer. A syscall log cannot.

This principle has several concrete consequences:

- **No ML in the agent.** Guardian contains no machine learning model, no embedding, no similarity score. It records numbers: PIDs, UIDs, syscall numbers, byte counts, file paths, network addresses.
- **No rule evaluation beyond local alerts.** The only logic Guardian applies locally is a small set of immediate-action alerts (sandbox escape detection, unexpected network connections). These exist because some events require a response faster than a round-trip to the platform. Everything else — anomaly detection, policy evaluation, risk scoring — happens in the platform.
- **No interpretation of file content.** Guardian records that a process called `read()` on a file descriptor that pointed to `/etc/passwd`. It does not read the file's contents, classify the read as suspicious, or decide whether the action was authorised.
- **No reports.** Guardian produces events, not reports. Reports are generated by the platform, which has the full organisational context needed to make them meaningful.

The "dumb by design" principle also means Guardian is auditable. Its entire decision surface is: which processes to watch, which syscalls to capture, and whether an event triggers a local alert. Every one of those decisions is expressed in a single human-readable YAML file.

## Who Uses Guardian

### Customer DevOps / MLOps Teams

These are the people who deploy and operate Guardian. Their interaction with Guardian consists of:

1. Writing or receiving a `guardian.yaml` file that maps process names to model names and specifies which syscalls to capture.
2. Running `python -m agent.main --config guardian.yaml` (or deploying the DaemonSet in Kubernetes).
3. Watching the rich terminal output to confirm events are flowing.
4. Occasionally checking buffer status if the control plane is unreachable.

They do not need to understand the hash chain, the gRPC protocol, or the EU AI Act. Guardian is designed so that a competent DevOps engineer can deploy it in under 30 minutes without reading anything beyond the installation guide.

### Compliance Officers / Auditors

These are the people who consume Guardian's output — but they never interact with Guardian directly. They work in the Viriato web interface, which presents the event stream as structured timelines, compliance dashboards, and exportable audit reports.

From an auditor's perspective, Guardian is the source of provenance: events that reach the platform were signed by an agent running on the same host as the AI model, at the time the syscall occurred. The hash chain means an auditor can verify that a record has not been altered after the fact.

### Security Researchers / Red Teams

These users may interact with Guardian's lower layers: reading the BPF program, inspecting the proto schema, calling `Signer.verify_chain()` directly, or analysing the buffer file at `~/.guardian/buffer/pending.jsonl`. The codebase is written to be readable at this level.

## What Guardian Produces

Guardian produces a continuous stream of **signed events**. Each event is a `RawEvent` dataclass with 16 fields in three layers:

**Kernel layer** (filled by eBPF or the fake generator):
- `timestamp` — nanosecond-precision UTC timestamp
- `pid` — process ID
- `process` — process name (up to 16 characters, from the kernel `TASK_COMM_LEN` equivalent)
- `syscall` — syscall name (e.g., `read`, `openat`, `connect`)
- `fd_path` — file descriptor path (up to 256 characters)
- `bytes` — byte count for I/O operations
- `network_addr` — network address string (up to 64 characters)
- `return_val` — syscall return value
- `uid` — user ID of the calling process

**Enrichment layer** (filled by `agent/enricher.py`):
- `agent_id` — UUID identifying this Guardian installation
- `model_name` — AI model name, resolved from the process name via `guardian.yaml`
- `container_id` — 12-character Docker short container ID, parsed from `/proc/<PID>/cgroup`
- `pod_name` — Kubernetes pod name from the `KUBERNETES_POD_NAME` environment variable
- `namespace` — Kubernetes namespace from the `KUBERNETES_NAMESPACE` environment variable

**Chain layer** (filled by `agent/signer.py`):
- `prev_hash` — SHA-256 hash of the previous event (or 64 zeros for the genesis event)
- `this_hash` — SHA-256 hash of this event's content

These events are batched, further signed at the batch level with HMAC-SHA256, and sent via gRPC to the Viriato control plane.

## What Guardian Does NOT Do

Understanding Guardian's non-goals is as important as understanding its goals. The following capabilities are explicitly out of scope for the agent:

| What it does not do | Where it happens instead |
|---------------------|--------------------------|
| Anomaly detection / ML scoring | viriato-platform |
| Risk assessment or policy evaluation | viriato-platform |
| Compliance report generation | viriato-web |
| Reading file contents | Not in scope at any layer |
| Intercepting network payloads | Not in scope at any layer |
| Modifying AI model behaviour | Not in scope at any layer |
| Authentication or authorisation of AI requests | AI framework / API gateway layer |
| Long-term storage of events | viriato-platform |
| User-facing dashboards | viriato-web |

If you are looking for a tool that does anomaly detection, generates compliance reports, or integrates with your SIEM, those capabilities exist in the platform tier — not in Guardian itself.

## The Three-Component System

Guardian is one component of a three-component product:

```
┌─────────────────────────────────────────────────────────────────┐
│  Customer Infrastructure                                        │
│                                                                 │
│   ┌──────────────┐         ┌─────────────────────────────────┐  │
│   │  AI Model    │         │  Guardian Agent                 │  │
│   │  (python /   │ ──────► │  eBPF + enricher + signer       │  │
│   │  torchserve) │ syscall │  (this repository)              │  │
│   └──────────────┘         └──────────────┬────────────────┘  │
│                                           │ gRPC / TLS         │
└───────────────────────────────────────────┼─────────────────────┘
                                            │
                              ┌─────────────▼──────────────┐
                              │  viriato-platform           │
                              │  Event storage, indexing,   │
                              │  anomaly detection, policy  │
                              │  evaluation, compliance API │
                              └─────────────┬──────────────┘
                                            │
                              ┌─────────────▼──────────────┐
                              │  viriato-web                │
                              │  Compliance dashboards,     │
                              │  audit timelines, reports,  │
                              │  alerts UI                  │
                              └────────────────────────────┘
```

**Guardian** runs on the customer's infrastructure, as close to the kernel as possible. It has no dependencies on the platform being reachable — events are buffered locally if the connection is lost.

**viriato-platform** is a Viriato-operated (or self-hosted) service that receives the event stream, stores it, indexes it, and runs the interpretation layer. It is where the "intelligence" lives: anomaly detection, policy engines, compliance mappings.

**viriato-web** is the human interface. Compliance officers, auditors, and security teams interact with the event data here. They see timelines, summaries, risk scores, and exportable reports — but they are always looking at derived views of the raw, signed event stream that Guardian produced.

This separation is intentional. See [Solution Architecture](solution-architecture.md) for the full rationale.

## The Minimal Deployment

The simplest possible Guardian deployment requires only:

```yaml
# guardian.yaml
agent:
  token: "your-token-here"
  control_plane: "ingest.viriato.io:443"
  batch_interval_ms: 100
  buffer_path: "~/.guardian/buffer"

watch:
  - process: "python"
    model_name: "my-model-v1"

syscalls:
  - read
  - write
  - openat
  - execve
  - connect
```

Run with:

```bash
python -m agent.main --config guardian.yaml
```

That is the complete deployment surface. No SDK installation in the AI model process, no environment variable injection into the model's runtime, no sidecar container (though sidecar deployment is supported), no kernel module compilation.

## Design Constraints

Guardian was designed under three hard constraints that shaped every architectural decision:

1. **Cannot modify AI model code.** AI models are often third-party, packaged, or contractually immutable. Any observability solution that requires model code changes cannot be universally deployed.

2. **Must work with any framework.** PyTorch, TensorFlow, TorchServe, vLLM, Hugging Face Inference Endpoints, custom Flask APIs — Guardian does not know or care what framework the model uses. It watches the process, not the application.

3. **Must not add latency.** eBPF programs run in the kernel's context with near-zero overhead. Guardian does not intercept syscalls (which would add latency) — it observes them asynchronously via a ring buffer.
