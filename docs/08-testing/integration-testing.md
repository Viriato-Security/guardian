# Integration Testing

Integration tests verify that Guardian's modules work together correctly and that the agent communicates with external systems as expected. Unlike unit tests, integration tests may write to the filesystem, spawn subprocesses, and use real gRPC communication.

Integration tests do not require root, BPF, or a real control plane endpoint. Everything is exercised locally.

---

## What Integration Tests Cover

The integration layer sits between the 63 unit tests and the manual end-to-end demo. It answers questions that unit tests cannot:

- Does a signed batch actually arrive at a gRPC server?
- Does the batch signature match what the server receives?
- Does the disk buffer grow when the server is unreachable?
- Does the buffer drain in FIFO order when the server returns?
- Does the full pipeline — generator → enricher → signer → sender — produce valid output?

## The gRPC Test Server

The core integration testing primitive is the embedded gRPC test server defined in `tools/demo.py`. The server code is stored as the string constant `_TEST_SERVER_CODE` and is written to a temporary Python file then launched as a subprocess:

```python
_TEST_SERVER_CODE = """\
import sys, grpc
from concurrent import futures
sys.path.insert(0, {repo_root!r})
from proto import guardian_pb2, guardian_pb2_grpc

class _Svc(guardian_pb2_grpc.GuardianIngestServicer):
    def StreamEvents(self, request_iterator, context):
        total = 0
        for batch in request_iterator:
            n = len(batch.events)
            total += n
            sig = batch.signature[:16] if batch.signature else "nosig"
            print(f"BATCH:{n}:{sig}", flush=True)
        return guardian_pb2.Ack(received=True, events_stored=total)

_srv = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
guardian_pb2_grpc.add_GuardianIngestServicer_to_server(_Svc(), _srv)
_srv.add_insecure_port('[::]:50051')
_srv.start()
print('SERVER_READY', flush=True)
_srv.wait_for_termination()
"""
```

The server:
1. Listens on `localhost:50051` with an insecure (no-TLS) channel
2. For each incoming `EventBatch`, prints `BATCH:{n}:{sig}` to stdout where `n` is the event count and `sig` is the first 16 characters of the batch signature
3. Prints `SERVER_READY` once the port is bound and ready to accept connections
4. Returns an `Ack` with `received=True` and the total event count

The `_start_test_server()` function writes the code to a `tempfile.NamedTemporaryFile`, launches it with `subprocess.Popen`, and reads stdout until `SERVER_READY` appears (with a 6-second timeout):

```python
def _wait_server_ready(proc: subprocess.Popen, timeout: float = 6.0) -> bool:
    deadline = time.monotonic() + timeout
    while time.monotonic() < deadline:
        line = proc.stdout.readline()
        if "SERVER_READY" in line:
            return True
        if proc.poll() is not None:
            return False
        time.sleep(0.05)
    return False
```

### What to Verify Against the Test Server

An integration test that uses the gRPC test server should verify all of the following:

**1. Batch delivery**

The server output must contain at least one `BATCH:N:sig` line after the agent runs, confirming that at least one batch was received.

**2. Signature format**

The 16-character prefix printed by the server comes from a 64-character hex HMAC-SHA256 batch signature. A full integration test should verify the complete 64-character signature is valid hex:

```python
assert len(signature) == 64
int(signature, 16)  # raises ValueError if not valid hex
```

**3. Event count**

The `N` value in `BATCH:N:sig` must match the number of events generated by the agent in the same interval. For a single-batch test, this is the batch size configured in `guardian.yaml`.

**4. Chain verification**

After events arrive at the server, deserialise them from the proto and call `verify_chain(events)`. The chain must pass, confirming that `prev_hash` / `this_hash` links are intact end-to-end through serialisation and deserialisation.

## Disk Buffer Integration Test

The disk buffer integration test verifies three phases:

### Phase 1: Buffer Growth

Start the agent with `GUARDIAN_FAKE_EVENTS=1` pointing at a non-existent server (e.g., `localhost:59999`). After several batch intervals, assert:

```python
buffer_file = Path("~/.guardian/buffer/pending.jsonl").expanduser()
assert buffer_file.exists()
lines = buffer_file.read_text().splitlines()
assert len(lines) >= 1
```

Each line should be valid JSON with the structure:

```json
{"agent_id": "...", "signature": "...", "events": [{...}, ...]}
```

Validate the structure of at least one line:

```python
batch = json.loads(lines[0])
assert "agent_id" in batch
assert "signature" in batch
assert isinstance(batch["events"], list)
assert len(batch["events"]) > 0
```

### Phase 2: Server Becomes Available

Start the gRPC test server. On the next successful send, the agent calls `_drain_buffer()` which replays batches in FIFO order.

### Phase 3: Buffer Drain

After a successful drain cycle, the buffer file should be empty or non-existent:

```python
if buffer_file.exists():
    assert buffer_file.read_text().strip() == ""
```

The server output should show BATCH lines whose total event count matches the sum of events across all buffered batches.

### 10,000 Line Cap

To verify the cap, fill the buffer to 10,000 lines programmatically (or configure a tiny batch size and run long enough), then verify that:

- Lines beyond 10,000 are dropped with a `WARNING: Disk buffer full (10000 lines) — dropping batch` log message
- The file does not grow beyond 10,000 lines

## tools/demo.py as Integration Test

`tools/demo.py` is the primary end-to-end integration vehicle for Phase 1. Its 10 scenes exercise the full pipeline:

| Scene | What it tests |
|-------|--------------|
| 1 | Architecture overview (no code execution) |
| 2 | `FakeEventGenerator` producing valid `RawEvent` instances |
| 3 | `Enricher` adding agent_id, model_name, pod metadata |
| 4 | `Signer` building the hash chain and verifying it |
| 5 | `LocalAlertEngine` detecting sandbox escapes and unexpected network |
| 6 | `Sender` sending to the embedded gRPC test server |
| 7 | Disk buffer: buffering when server is down, draining when it returns |
| 8 | Full pipeline running in a thread with live event table |
| 9 | Chain tamper detection |
| 10 | Summary metrics |

The demo uses real code paths, not mocks. Scene 6 starts the actual test server subprocess and verifies that `BATCH:N:sig` lines appear in its stdout. This constitutes an integration test of the gRPC transport.

Run the full demo:

```bash
python tools/demo.py
```

Run a specific scene:

```bash
python tools/demo.py --scene 6
```

Run with minimal delays (for CI-style automated verification):

```bash
python tools/demo.py --speed fast
```

## Pipeline Integration Test

A pipeline integration test exercises the full `main.py` startup sequence in dry-run mode:

```bash
GUARDIAN_FAKE_EVENTS=1 python -m agent.main \
    --config guardian.yaml \
    --dry-run \
    --log-level DEBUG
```

Expected output includes:

```
DEBUG agent.config    - Loaded config from guardian.yaml
DEBUG agent.enricher  - Agent ID: <uuid>
DEBUG agent.signer    - Signed event 1
DEBUG agent.sender    - Dry-run: skipping gRPC send
```

Verify in CI by running for 2–3 batch intervals and asserting the process exits cleanly:

```bash
timeout 5 python -m agent.main --fake --dry-run --log-level DEBUG
# Exit code 0 or 124 (timeout) are both acceptable
```

A non-zero exit code other than 124 (timeout kill) indicates a pipeline error.

## Environment Variables for Integration Tests

| Variable | Purpose |
|----------|---------|
| `GUARDIAN_FAKE_EVENTS=1` | Force fake event generator; no BPF required |
| `GUARDIAN_INSECURE_GRPC=1` | Use insecure gRPC channel for local test server |

Both variables are required for integration tests that run without a TLS certificate and without a real eBPF probe.

## Phase 2 Integration Testing

When `EbpfLoader` ships in Phase 2, the integration test suite expands:

### eBPF Loader Tests

- Verify that `EbpfLoader.is_available()` returns `True` on Linux with BPF enabled
- Verify that events emitted by `EbpfLoader` pass all 16 tests in `test_generator.py`
- Verify that `RawEvent` field types match between `FakeEventGenerator` and `EbpfLoader`

### Real Syscall Tests

Run a known workload (e.g., `open("/tmp/test.txt")`, `write(fd, data)`) and verify the corresponding events appear in the pipeline:

```python
import subprocess
import time

# Start Guardian with eBPF
agent = subprocess.Popen(["sudo", "python", "-m", "agent.main", "--log-level", "DEBUG"])

# Generate known syscall
with open("/tmp/test_guardian.txt", "w") as f:
    f.write("test")

time.sleep(0.5)  # Allow one batch interval

# Verify event in buffer or server output
```

### OrbStack for macOS Development

Phase 2 eBPF integration tests require Linux. On macOS, use OrbStack to run a lightweight Ubuntu VM:

```bash
orb create ubuntu:22.04 guardian-test
orb push guardian-test /path/to/guardian /home/user/guardian
orb run guardian-test bash install.sh
orb run guardian-test sudo pytest tests/integration/ -v
```

OrbStack provides full Linux kernel access (including BPF) without the overhead of a full VM.
